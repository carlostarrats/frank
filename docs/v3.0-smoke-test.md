# Frank v3.0 Smoke Test

Run-through checklist for validating v3.0 end-to-end before tagging. Consolidated from Phases 1–5 + the v2 image/PDF share fix. Designed for one session, ~45 minutes.

**How to use:** Work top-to-bottom. Tick `- [x]` each step as you verify. If a step fails, jot observed behavior in the **Bug log** at the bottom and continue — many later steps are independent. Don't stop at the first failure; the goal is a complete picture.

**When done:** decide tag-ready (all ticked, empty bug log) or tag-blocked (anything in bug log with higher severity than cosmetic).

## 2. Canvas live share (Phase 2)

### Happy path

- [ ] Reuse the canvas project from Section 1. Click Share → "Start live share" in the popover.
- [ ] Daemon console prints something like `[live-share] <shareId> canvas → state (first push)` within ~1 second.
- [ ] In second browser, open the share URL. Canvas renders with current state + comments.
- [ ] In author app: add a new shape. Viewer updates within ~1–2 seconds.
- [ ] Drag an existing shape. Viewer updates in near-real-time (subject to 15/s rate cap).
- [ ] Drop a NEW image onto the canvas. Viewer picks up the image within ~1–2 seconds.
- [ ] Wait ~30 seconds without editing, then make one edit. Daemon console prints `→ state (30s since last state)`. Prior edits should have shown `→ diff`.

### Pause / Resume

- [ ] Click "Pause live share" in popover. Author popover shows "Resume live share".
- [ ] In viewer: "Author offline" or "Author ended live share" indicator appears.
- [ ] Edit the canvas. Viewer does NOT update.
- [ ] Click "Resume live share". Edit again. Viewer picks up the new edit within ~1–2 seconds.

### Bandwidth cap (heavy canvas)

- [ ] Create a fresh canvas project. Drop 3 large images (3–5 MB each, total > 10 MB bundled).
- [ ] Start live share. Make rapid edits for ~30 seconds.
- [ ] At some point the share popover should flip to "Live updates throttled — catching up" (sticky banner, not a toast).
- [ ] Stop editing for ~15 seconds. Banner clears, status returns to "Live · N watching".

### Oversized-payload (413) handling

- [ ] Create a canvas project with a single huge image (≥ 2 MB raw, > 1 MB base64). Start live share.
- [ ] First push hits 413. Popover flips to "paused" state with the content-type-aware banner: "Canvas too heavy for live share — reduce inline assets, then click Resume."
- [ ] Remove the big image, click Resume. Live share resumes normally.

## 3. Image live share (Phase 3)

### Viewer opens BEFORE live starts

- [ ] Fresh image project, 2 pin comments, create share, click "Start live share".
- [ ] Open share URL in second browser. Image renders with both comments in sidebar.
- [ ] Author: add a comment. Viewer picks it up within ~1 second.
- [ ] Author: delete a comment. Viewer removes it.
- [ ] Author: approve a comment via feedback panel. Viewer reflects the status change.
- [ ] Daemon console: first push `→ state`, subsequent `→ diff`, eventual `→ state (30s since last state)` after idle.

### Cold-open MID-session (the harder path)

- [ ] Start a new image project. Create share. Start live share.
- [ ] Add 5 comments over ~25 seconds (console shows state then diffs).
- [ ] Wait for (or trigger via one more comment after 30s idle) a `→ state (30s since last state)` line.
- [ ] Add 2–3 more comments (these go `→ diff`).
- [ ] NOW open share URL in second browser for the first time.
- [ ] Verify: image renders; ALL comments from steps 3–5 are visible (not just the early ones).
- [ ] Add one more comment in author app. Viewer picks up within ~1 second.

If the cold-opener's comment list is missing the diffs (stale): they aged out of the rolling buffer AND the snapshot wasn't refreshed in time. Note in bug log — this is a state-promotion timing issue.

## 4. PDF live share (Phase 4a)

### Viewer opens BEFORE live starts

- [ ] PDF project with 1–2 comments. Create share. Click "Start live share".
- [ ] Open share URL in second browser. PDF renders in iframe; comments in sidebar.
- [ ] Add a pin comment on page 1 from author app. Viewer comment list updates.
- [ ] Add a pin on page 2. Viewer comment list updates (note: page sync is NOT in v3.0 — the viewer's PDF iframe does NOT jump to page 2 automatically; that's Phase 4b scope).

### Cold-open MID-session

- [ ] Repeat the pattern from Section 3's cold-open test, substituting PDF.
- [ ] Verify all mid-session comments appear on the viewer's first load.

### Graceful degradation (the architectural proof)

- [ ] With live PDF share active + viewer tab open:
  - [ ] Pause live share from the author popover.
  - [ ] Viewer tab: PDF iframe **keeps rendering the PDF** (this is the critical check — validates that `renderPdfLive` never touches the iframe).
  - [ ] Viewer comment list stops updating — new author comments don't appear.
  - [ ] Resume live share. Comment updates resume. PDF iframe never reloaded.

### Cross-browser PDF rendering (v2-inherited)

- [ ] Open the same PDF share URL in TWO browsers (Chrome + one other).
- [ ] If PDF renders correctly in Chrome but NOT in Firefox/Safari: **NOT a Phase 4a bug.** File in bug log as "v2 inherited: data-URL PDF iframe rendering in {browser}." Doesn't block v3.0 tag — was present in v2 fix too, to be addressed alongside Phase 4b's PDF.js migration.

## 5. Phase 5 lifecycle + presence polish

### Revoke flow

- [ ] Any live-share-active project. Open popover. Click "Revoke share".
- [ ] Native confirm dialog appears with multi-line text: link stops working + project is unchanged + can create new share.
- [ ] Click OK. Daemon broadcasts `share-revoked`.
- [ ] Viewer tab: "This live share has ended" overlay appears; page becomes inert.
- [ ] Author tab: popover returns to "Create share" state (link input cleared).
- [ ] Toolbar's ambient LIVE badge clears.
- [ ] Create a new share. Original link still dead (try it in a third browser — should 410).

### Expiration picker

- [ ] Fresh project, open share modal. Select "1 day" from expiration dropdown. Create share.
- [ ] Check `~/.frank/projects/<projectId>/project.json`:
  ```bash
  cat ~/.frank/projects/*/project.json | grep -A 1 activeShare
  ```
  `activeShare.expiresAt` should be ~24 hours from now. If it's ~7 days, the expiration picker didn't flow through (bug log).
- [ ] Create another share, pick "90 days". Verify `expiresAt` is ~90 days out.
- [ ] Create a default share (don't touch the select). Verify `expiresAt` is ~7 days out.

### Content-type-aware banner copy

- [ ] Force a 413 on each project type:
  - Canvas with large inline images → "Canvas too heavy for live share — reduce inline assets..."
  - Image project with a 2 MB+ photo → "Image too large for live share — use a smaller file..."
  - PDF project with a 2 MB+ PDF → "PDF too large for live share — use a smaller file..."
- [ ] Each project type should show its OWN wording, not the canvas default.

### Ambient LIVE badge

- [ ] Start live share on a canvas project. Close the popover.
- [ ] Toolbar share button shows a small pulsing red "LIVE · N" badge.
- [ ] Open a second viewer tab. Badge updates to "LIVE · 2".
- [ ] Close the viewer tab. Count decreases.
- [ ] Pause live share. Badge disappears.
- [ ] Resume. Badge returns.
- [ ] Switch to a different project (no live share). Badge should not appear.
- [ ] Switch back to the live-sharing project. Badge reappears immediately — not after a 30-second delay. (This is the `syncToolbarLiveBadge` post-mount fix.)
- [ ] Repeat for an image project AND a PDF project — both have their own share buttons in different views; both should show the badge.

### 2-hour auto-pause (fast-forward with env override)

- [ ] Stop the daemon. Restart with a 1-minute session max:
  ```bash
  FRANK_DEBUG_LIVE_SHARE=1 FRANK_SESSION_MAX_MS=60000 frank start
  ```
- [ ] Start live share on any project. Wait ~65 seconds without editing.
- [ ] Popover banner should appear with verbatim copy:
  > Live share paused — sessions auto-pause after 2 hours to prevent accidental long-running sessions. Click Resume to continue.
- [ ] Click Resume. Fresh session clock starts (another 60 seconds until next auto-pause).
- [ ] Restart daemon WITHOUT the env override to restore normal 2h default.
